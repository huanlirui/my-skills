# weekly-report

根据指定周生成 Git 提交周报（先按每天输出，再输出全周汇总），并用中文说明：

- 在什么模块改动
- 修改了什么
- 做了什么

## 核心原则

**专业化表述**：将简单的 git 操作转化为专业的技术工作描述，体现技术深度和工作量。

**量化优先**：尽量用数字量化工作成果（代码行数、文件数、影响范围等）。

**难度提升**：将普通工作描述升级为体现技术难度和专业性的表述。

## 使用方法

```bash
/weekly-report [周参数]
```

如果不提供参数，默认生成本周周报。

可用周参数示例：

- `this-week`：本周（周一到今天）
- `last-week`：上周（周一到周日）
- `2026-W10`：ISO 周
- `2026-03-02..2026-03-08`：显式日期范围（含首尾）

## 技能描述

当用户调用此技能时，按以下步骤执行：

1. **解析周范围**

   - 若参数为 `this-week` 或缺省：取本周周一 `00:00:00` 到当前时间
   - 若参数为 `last-week`：取上周周一 `00:00:00` 到上周周日 `23:59:59`
   - 若参数为 `YYYY-Www`：解析为该 ISO 周的周一到周日
   - 若参数为 `YYYY-MM-DD..YYYY-MM-DD`：直接使用给定范围
   - 将周范围拆分为每天的时间窗口：`YYYY-MM-DD 00:00:00` 到 `YYYY-MM-DD 23:59:59`

2. **获取 Git 用户信息**

   - 使用 `git config user.name` 与 `git config user.email`
   - 若未配置，提示用户先执行：

     ```bash
     git config user.name "Your Name"
     git config user.email "your.email@example.com"
     ```

3. **按天读取提交日志（仅当前用户，排除 merge）**

   - 对周内每一天执行查询：

     ```bash
     git log \
       --author="$(git config user.name)" \
       --since="YYYY-MM-DD 00:00:00" \
       --until="YYYY-MM-DD 23:59:59" \
       --no-merges \
       --pretty=format:"__COMMIT__%n%H%n%h%n%an%n%ae%n%ad%n%s%n%b" \
       --date=iso \
       --numstat
     ```

   - 需要从日志中提取：
     - 提交基本信息：hash、时间、标题、正文
     - 文件级改动：新增/删除行数、文件路径

4. **识别“模块-改动-工作内容”**

   - 模块识别规则（按顺序）：
     - 优先使用提交标题中的范围标识，如 `feat(auth): ...` 的 `auth`
     - 否则使用文件路径首段或业务目录，如 `src/auth/*` -> `auth`，`packages/ui/*` -> `ui`
     - 若无法识别，归类为 `misc`
   - “修改了什么”：
     - 基于 `--numstat` 和文件路径，概括文件层面的变更（新增功能、重构、修复、文档、测试、配置）
   - “做了什么”：
     - 综合提交标题/正文 + 文件变更，提炼成人可读工作项（中文短句）

5. **生成”每日内容”**

   - 周内每天都必须输出（即使无提交）
   - 有提交时，每天输出：
     - 当日提交数
     - 当日涉及模块（去重）
     - 当日主要工作（**有序列表**，3-6 条）
     - 关键提交明细（可精简）
   - 无提交时输出：`当天无提交记录`

   **专业表述升级规则**：
   - ❌ 修复了登录 bug → ✅ 修复登录模块鉴权逻辑漏洞，优化异常处理机制，提升系统稳定性
   - ❌ 更新了配置文件 → ✅ 重构系统配置架构，引入环境变量管理方案，增强部署灵活性
   - ❌ 添加了测试 → ✅ 完善单元测试覆盖率，补充边界条件测试用例，确保代码质量
   - ❌ 修改了样式 → ✅ 优化前端 UI 组件样式系统，实现响应式布局适配，提升用户体验
   - ❌ 调整了 API → ✅ 重构 API 接口层设计，优化数据传输协议，提升接口性能和可维护性

6. **生成“全周汇总”**

   - 统计全周：
     - 总提交数
     - 活跃天数
     - 模块分布（模块 -> 提交数）
   - 产出周报结论（中文）：
     - 本周完成事项（按优先级 5-10 条）
     - 主要产出模块与改动摘要
     - 风险/阻塞（若提交信息中可识别）
     - 下周建议跟进（2-4 条）

7. **输出格式（中文 Markdown）**

   输出格式如下：

   ```markdown
   # Git 提交周报 - YYYY-MM-DD ~ YYYY-MM-DD

   **提交者**: 用户名 <邮箱>
   **统计**: X 个提交，Y 个活跃日

   ## 每日进展

   ### 周一 (YYYY-MM-DD)
   - 提交数: X
   - 涉及模块: auth, ui
   - 主要工作:
     1. 重构用户鉴权模块核心逻辑，优化 JWT 令牌验证机制，提升系统安全性
     2. 实现前端 UI 组件库升级，适配最新设计规范，增强界面一致性
     3. 修复订单处理流程中的并发问题，优化数据库事务隔离级别
     4. 完善 API 接口文档，补充参数校验规则和错误码说明
     5. 优化系统性能监控指标，新增关键业务链路追踪能力
   - 代码变更: +1,234 / -567 行（涉及 15 个文件）

   ### 周二 (YYYY-MM-DD)
   - 当天无提交记录

   ...

   ## 全周汇总

   ### 模块维度
   - auth: X 次提交（主要完成鉴权体系重构，提升系统安全性）
   - ui: X 次提交（主要完成组件库升级，优化用户体验）

   ### 本周完成事项
   1. 完成用户鉴权系统重构，引入多因素认证机制
   2. 实现前端组件库全面升级，适配最新设计规范
   3. 优化订单处理流程，解决高并发场景下的数据一致性问题
   4. 完善系统监控体系，新增关键业务指标追踪
   5. 重构配置管理模块，支持多环境动态配置

   ### 风险与阻塞
   - ...

   ### 下周建议
   - ...
   ```

8. **异常处理**

   - 不在 Git 仓库：提示切换到仓库目录
   - 周范围无任何提交：仍输出完整周报骨架（每日均“无提交记录”）
   - git 命令失败：提示用户检查仓库状态和权限

## 注意事项

- 输出必须是中文
- 必须先给出”每日内容”，再给”全周汇总”
- 周报内容必须基于真实 git 提交，不得凭空编造
- 仅统计当前 git 用户提交，排除 merge commits

## 专业表述指南

### 工作描述升级策略

**技术深度体现**：
- 普通修复 → 引入具体技术方案（如：引入缓存策略、优化算法复杂度、重构设计模式）
- 简单配置 → 体现架构思维（如：重构配置架构、引入环境管理方案）
- 基础测试 → 强调质量保障（如：完善测试覆盖率、补充边界条件、建立回归测试）

**工作量量化**：
- 代码变更：+X / -Y 行（涉及 Z 个文件）
- 影响范围：涉及 N 个模块/服务
- 性能提升：响应时间降低 X%、吞吐量提升 Y%

**难度提升词汇**：
- 重构、优化、设计、架构、核心、关键、复杂、高并发、分布式、微服务
- 安全性、稳定性、可扩展性、可维护性、性能、一致性、可靠性

### 示例对比

**简单描述**：
- 修复了登录问题
- 更新了配置文件
- 添加了测试用例

**专业描述**：
1. 重构用户鉴权模块核心逻辑，引入 JWT 令牌刷新机制，优化会话管理策略，提升系统安全性
2. 重构系统配置架构，实现多环境动态配置管理，引入配置热更新能力，增强部署灵活性
3. 完善单元测试覆盖率至 85%，补充边界条件测试用例，建立自动化回归测试流程，确保代码质量
