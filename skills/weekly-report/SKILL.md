# weekly-report

根据指定周生成 Git 提交日报，格式为：日期 + 编号工作项（专业表述）。

## 核心原则

**专业化表述**：将简单的 git 操作转化为专业的技术工作描述，体现技术深度和工作量。

**量化优先**：尽量用数字量化工作成果（代码行数、文件数、影响范围等）。

**难度提升**：将普通工作描述升级为体现技术难度和专业性的表述。

## 使用方法

```bash
/weekly-report [周参数]
```

如果不提供参数，默认生成本周周报。

可用周参数示例：

- `this-week`：本周（周一到今天）
- `last-week`：上周（周一到周日）
- `2026-W10`：ISO 周
- `2026-03-02..2026-03-08`：显式日期范围（含首尾）

## 执行步骤

1. **解析周范围**

   - 若参数为 `this-week` 或缺省：取本周周一到当前时间
   - 若参数为 `last-week`：取上周周一到上周周日
   - 若参数为 `YYYY-Www`：解析为该 ISO 周的周一到周日
   - 若参数为 `YYYY-MM-DD..YYYY-MM-DD`：直接使用给定范围

2. **获取 Git 用户信息**

   - 使用 `git config user.name` 与 `git config user.email`
   - 若未配置，提示用户先执行：

     ```bash
     git config user.name "Your Name"
     git config user.email "your.email@example.com"
     ```

3. **按天读取提交日志（仅当前用户，排除 merge）**

   对周内每一天执行查询（**必须搜索所有分支**，不限于当前分支）：

   ```bash
   git log \
     --all \
     --author="$(git config user.name)" \
     --since="YYYY-MM-DD 00:00:00" \
     --until="YYYY-MM-DD 23:59:59" \
     --no-merges \
     --pretty=format:"__COMMIT__%n%s%n%b" \
     --numstat
   ```

4. **提炼工作项**

   - 综合提交标题、正文和文件变更，提炼成人可读的中文工作项
   - 同一天的多条提交按逻辑归类合并，避免重复
   - 如果一天有多条不同工作，拆分为多条编号项
   - 无提交的日期跳过，不输出

   **工作项条数控制**：

   - 每日工作项尽可能不少于 4 条
   - 如果归纳后不足 4 条，适当拆分：一个较大的工作项可以按"做了什么"+"达到什么效果"拆成两条
   - **重要**：即使提交记录很少（如只有 1-2 条），也必须如实输出，不要因为条数不足而忽略或跳过
   - 每条工作项描述控制在一句话内，不要过长
   - 拆分示例：
     - 合并时：完善延期审核申请单位接口参数修改，添加 tabType 参数以区分已完成和待办状态，并统一状态字段命名
     - 拆分后：
       1. 完善延期审核申请单位接口参数，添加 tabType 参数区分已完成和待办状态
       2. 统一延期审核状态字段命名，将 currStatus 替换为 processStatus

   **commit 信息不规范时的处理（重要）**：

   当提交标题过于简短、模糊（如"fix"、"update"、"tmp"、"wip"、纯英文无意义标题）时，不能直接翻译标题，必须通过以下方式推断实际工作内容：

   1. **看文件路径**：通过 `--numstat` 输出的文件路径判断涉及的业务模块（如 `src/views/audit/` → 延期审核模块）
   2. **看代码变更**：读取具体改动的文件内容，理解实际做了什么（新增了字段？改了接口？修了 bug？）
   3. **看提交正文**：有些 commit body 里有详细描述，即使标题很水
   4. **结合上下文**：同一天多个提交之间通常有关联，综合判断整体工作内容

   目标：即使 commit 写的是 "fix" 或 "update"，输出的工作项也要能准确反映实际业务工作（如"完善延期审核申请单位接口参数修改，添加 tabType 参数以区分已完成和待办状态"）。

5. **输出格式**

   **严格按以下格式输出，不要添加额外的统计信息、模块分析、全周总结等：**

   ```
   M.DD
   1. 专业化表述工作项1
   2. 专业化表述工作项2
   3. 专业化表述工作项3

   M.DD
   1. 专业化表述工作项1
   2. 专业化表述工作项2
   ```

   规则：
   - 日期格式为 `M.DD`（如 4.27、5.13），不带星期、不带年份
   - 每天一个日期块，日期后换行直接跟编号列表
   - 工作项用 `1. 2. 3.` 有序列表，不用 `-` 无序列表
   - 每条工作项一句话，按"专业表述升级规则"描述
   - 日期块之间空一行
   - 不输出提交数、模块、代码变更行数、全周汇总、风险、下周建议等
   - 有提交的日期才输出，无提交的日期跳过

6. **异常处理**

   - 不在 Git 仓库：提示切换到仓库目录
   - 周范围无任何提交：输出"本周无提交记录"
   - git 命令失败：提示用户检查仓库状态和权限

## 注意事项

- 输出必须是中文
- 必须基于真实 git 提交，不得凭空编造
- 仅统计当前 git 用户提交，排除 merge commits

## 专业表述升级规则

将普通工作描述升级为体现技术深度和专业性的表述：

- ❌ 修复了登录 bug → ✅ 修复登录模块鉴权逻辑漏洞，优化异常处理机制，提升系统稳定性
- ❌ 更新了配置文件 → ✅ 重构系统配置架构，引入环境变量管理方案，增强部署灵活性
- ❌ 添加了测试 → ✅ 完善单元测试覆盖率，补充边界条件测试用例，确保代码质量
- ❌ 修改了样式 → ✅ 优化前端 UI 组件样式系统，实现响应式布局适配，提升用户体验
- ❌ 调整了 API → ✅ 重构 API 接口层设计，优化数据传输协议，提升接口性能和可维护性

**技术深度体现**：

- 普通修复 → 引入具体技术方案（如：引入缓存策略、优化算法复杂度、重构设计模式）
- 简单配置 → 体现架构思维（如：重构配置架构、引入环境管理方案）
- 基础测试 → 强调质量保障（如：完善测试覆盖率、补充边界条件、建立回归测试）

**难度提升词汇**：

- 重构、优化、设计、架构、核心、关键、复杂、高并发、分布式、微服务
- 安全性、稳定性、可扩展性、可维护性、性能、一致性、可靠性

**示例对比**：

- 简单描述：修复了登录问题
- 专业描述：修复登录模块鉴权逻辑漏洞，优化异常处理机制，提升系统稳定性

- 简单描述：更新了配置文件
- 专业描述：重构系统配置架构，引入环境变量管理方案，增强部署灵活性
